Implementation explainer

Bifrost now speaks Pi natively.

Issue #835 adds a first-party Pi package that launches Bifrost safely, discovers its MCP tools, presents them as native Pi tools, and ships the three canonical code-intelligence skills alongside them.

What changed

The existing plugins/bifrost-agent directory became a real Pi package. It still serves Codex, Claude, Amp, and other hosts. Pi is now another projection of that same package rather than a separate integration with copied skills and release logic.

01

A native Pi extension

The extension starts one Bifrost MCP process for each Pi session and registers Bifrost's advertised tools with Pi.

02

One trusted launcher path

Pi reuses the existing explicit-binary, managed-cache, opt-in PATH, and checksum-verified download policy.

03

Canonical skills

The package exposes only navigation, code reading, and codebase search. It points at the canonical skill directories instead of copying them.

04

Release-grade packaging

CI checks the package. Releases build an inspectable npm tarball. Automated npm publication remains deliberately separate.

The architecture at a glance

Pi sessionLoads the extension and canonical skills
Session controllerOwns state, tools, races, and cleanup
Trusted resolverSelects a verified Bifrost binary
MCP SDKOwns child stdio and protocol messages
BifrostRuns with explicit root and symbol|extended

The central boundary: the launcher decides which executable is trusted and how it must start. The MCP SDK owns the process and protocol after that. This keeps trust policy and transport lifecycle separate.

How a session starts

Pi loads the package

package.json names one extension and exactly three skill directories. Pi loads extensions/bifrost.ts when the package starts.

The extension captures Pi's workspace

The session_start handler passes ctx.cwd into the session controller. The controller does not infer the workspace from the installed package directory.

The launcher resolves a trusted binary

resolveBifrostLaunch validates the workspace and resolves Bifrost using the established policy. An explicit BIFROST_WORKSPACE_ROOT environment value cannot redirect this Pi session.

command: <resolved bifrost executable>
args:    --root <absolute Pi workspace> --mcp symbol|extended
cwd:     <absolute Pi workspace>

The official SDK opens stdio MCP

The MCP SDK starts Bifrost directly. It owns stdin and stdout. Bifrost diagnostics inherit stderr, which prevents log text from contaminating the JSON-RPC stream.

Pi gets native tool definitions

The controller calls tools/list, rejects duplicate or conflicting names, preserves each advertised JSON Schema, and registers the tools with Pi. Only then does the session become connected.

How a tool call moves through the bridge

Pi sees ordinary native tools such as get_summaries, search_symbols, and query_code. The bridge does not expose one generic “call MCP” tool.

Pi validates against Bifrost's schema

The adapter uses Type.Unsafe around the complete advertised JSON Schema. It preserves references, definitions, required fields, unions, and constraints instead of rebuilding a lossy TypeBox approximation.

The controller forwards the request

Arguments pass through unchanged. Pi's abort signal and a five-minute request timeout go to the MCP SDK.

The adapter maps the result

MCP error results become failed Pi tool executions. Text stays text. Images remain images. Structured content is serialized into model-visible text because Pi sends content, not details, to the model.

Large results remain usable

Model-visible text is limited to Pi's 2,000-line and 50 KB bounds. A truncation notice is added when needed. The complete MCP result remains available in details.

Why lifecycle handling needed its own controller

Starting a child process is easy. Keeping one correct child alive while Pi reloads, shuts down, or overlaps asynchronous starts is the harder part.

Generation checks stop stale starts

Every start or shutdown increments a generation counter. A slow earlier connection cannot replace a newer session after it finishes.

Cleanup is idempotent

closeOnce makes repeated shutdown paths safe. Active and still-starting clients are both tracked and closed.

Registered tools survive reload

Pi tool definitions remain registered, but their closures consult the current active client. A restart does not register duplicate definitions.

Dead connections fail closed

An unexpected MCP close clears active availability and reports an error. Existing tool names cannot silently continue against a dead client.

Choices made and tradeoffs considered

Extend the shared package

This keeps skills, versions, release metadata, and launcher policy in one place. A Pi-only package would have created another source of drift and another binary trust boundary.

Cost: The shared directory now has npm package concerns alongside other host projections.

Spawn the resolved binary directly

Pi asks the launcher module for a trusted command, then lets the MCP SDK own that child. It does not keep an extra launcher process between Pi and Bifrost.

Cost: The launcher needed a small typed programmatic API in addition to its CLI.

Use the official MCP SDK

The SDK handles initialization, stdio framing, requests, cancellation messages, and process teardown. A hand-built client would duplicate protocol machinery.

Cost: The package gains a pinned runtime dependency that must be reviewed and tested when upgraded.

Register discovered tools eagerly

Pi gets native names, descriptions, and schemas. The model can call query_code directly rather than going through a generic MCP proxy.

Cost: Session startup waits for Bifrost initialization and tool discovery.

Preserve JSON Schema wholesale

Type.Unsafe keeps Bifrost's schema authoritative. Reconstructing TypeBox definitions would lose uncommon JSON Schema features and create drift.

Cost: TypeScript cannot statically infer each runtime-discovered argument shape.

Limit the toolset

symbol|extended provides analyzer navigation and structural queries. Workspace switching and broad text/file tools stay out because Pi already owns the workspace and filesystem.

Cost: Bifrost tools outside those two sets are intentionally unavailable through this package.

Do not bundle Bifrost

The npm package remains small and follows one verified release policy. Local development can point at a debug binary with BIFROST_BINARY_PATH.

Cost: A first run may need a platform download and checksum verification.

Build, but do not auto-publish

Release CI creates a reviewable npm tarball. It does not assume npm credentials or trusted-publisher setup that the repository does not yet have.

Cost: Publishing to npm remains a separate maintainer action.

Other gates

TypeScript checks, package checks, isolated pi install, npm audit, pack and publish dry runs, workflow lint, generated-manifest checks, Rust formatting, and all-feature Clippy passed. A final independent code review found no blockers.

The feature-complete Rust suite passed 1,030 tests and ignored four. Three existing tests failed because their fixtures assume a master default branch while this environment initializes temporary repositories with main. This change contains no Rust source changes.

Known limitation: cancellation is prompt on Pi's side, not necessarily inside Bifrost

Pi's abort signal reaches the MCP SDK. The SDK rejects the pending Pi request and sends an MCP cancellation notification. The process-boundary test proves both behaviors.

Bifrost's current stdio server reads and executes one input line at a time. While analyzer work is running, that same loop cannot read the cancellation notification. The analyzer may therefore finish its current work before Bifrost observes cancellation.

The documentation says this plainly. The bridge does not claim that cancellation forcibly interrupts analyzer computation.

The end state

The implementation adds Pi without creating a parallel product. Pi uses the same canonical skills, release version, binary trust policy, and workspace rules as the other host integrations. The only Pi-specific code is the layer that Pi genuinely needs: session lifecycle, dynamic tool registration, and MCP-to-Pi type adaptation.
