# SQL-native analyzer: per-repo blob-keyed SQLite store replacing resident AnalyzerState

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds. It must be maintained in accordance with `.agent/PLANS.md` at the repository root.

## Purpose / Big Picture

Today bifrost's tree-sitter analyzer parses every file in a workspace at startup and keeps the extracted results — including every file's complete source text — in memory for the life of the process. On a large repository that is hundreds of megabytes to gigabytes of resident memory, startup re-parses whole branches' worth of files after a `git switch`, and every worktree of the same repository pays the full cost separately.

After this change, parse results live in a single SQLite database per primary repository, keyed by git blob OID (the SHA-1 content hash git itself assigns to a file's bytes). Any file content the analyzer has ever seen — on any branch, in any worktree of that repo — is parsed exactly once, ever. Queries pull the rows they need on demand instead of holding everything resident. Observable outcomes: switching branches and re-running analyzer queries causes zero re-parsing (provable with a parse counter); two worktrees of one repository share one cache; resident memory no longer scales with total workspace source size (provable with the memory benchmark added in Milestone 5); and all existing analyzer behavior tests still pass.

## Surprises & Discoveries

- Observation: The M1 store keeps a single `analysis_epoch` in `meta` and `open_for_workspace(language, ts_language)` derives the epoch from one language's grammar. In a multi-language workspace, per-language analyzers opening the shared DB with different epochs would each see a mismatch and wipe the whole database in turn (ping-ponging wipes on every startup). M2 must store the epoch per language (e.g. `meta` key `analysis_epoch:<lang>`) and scope invalidation to that language's rows only.
  Evidence: `src/analyzer/store/mod.rs` `migrate()` compares one `META_ANALYSIS_EPOCH` value and calls `recreate_schema` dropping all tables.
- Observation: `LiveSnapshot` is built from `.git/index` entries only, so untracked files are absent from `paths_for_oid`/`contains_oid`; workspace-wide queries would not report hits in brand-new files even after ingestion. Point resolution (`Liveness::oid_for_path`) does handle untracked paths. M2/M3 must feed the analyzer's own file enumeration (or a dirty/untracked overlay) into the snapshot.
  Evidence: `build_snapshot` in `src/analyzer/store/liveness.rs` iterates `repo.index()` entries only.
- Observation: Path-derived persisted content is broader than the Python case the plan called out. Audit results (M2a): Rust package names derive from the relative path; Go canonical packages combine `go.mod` content with the source package clause; JS/TS `contains_tests` has a filename component (`.test.`/`.spec.`) and JS/TS synthetic module units have filename-derived short names. All are excluded from storage and recomposed/synthesized at hydrate via the adapter hooks; Java/C++/C#/PHP/Scala/Ruby persisted content is source-derived.
  Evidence: `assert_round_trip` and `identical_python_blob_hydrates_with_live_path_names` in `src/analyzer/store/mod.rs` tests; per-adapter hook overrides in `src/analyzer/{python/adapter,go/adapter,rust/adapter,typescript/mod,javascript/mod}.rs`.
- Observation: The first M4a implementation synthesized module units in workspace queries by fully hydrating EVERY live file of the language on every `definitions()`/`search_definitions()` call — searchtools_service went from ~10s to 405s. Fixed by computing path-derived module names from live paths (adapters' existing naming code), pre-filtering by name before any DB work, and fetching the one content-derived bit synthesis needs (JS/TS "has structured imports") in a single chunked batched query. A permanent guard now asserts zero full hydrations for a search and a lookup over a 20-file fixture (hydration counter on the analyzer).
  Evidence: searchtools_service 405s → ~9s wall; `full_hydration_count` in `src/analyzer/tree_sitter_analyzer.rs`; guard test in `tests/analyzer_sql_query_parity.rs`. Lesson: in a pull-per-query design, any helper that iterates `all_paths()` and fetches per file is O(workspace) per query — wall-clock deltas in unchanged suites are the detection mechanism.
- Observation: `cargo clippy-no-cuda` is unusable from a worktree nested under the primary checkout (`.claude/worktrees/*`): cargo's config walk finds both the worktree's and the parent repo's `.cargo/config.toml` and concatenates the duplicate alias arrays, producing a garbled command line. The expanded equivalent (`cargo clippy --all-targets --features nlp,python -- -D warnings`) is what must be run in this worktree.
  Evidence: both config files define identical `clippy-no-cuda` arrays; alias expansion doubled the arguments and clippy-driver rejected them.
- Observation: The resident-free lookup-index build must not hydrate every live file. Building it from `code_units` rows with `in_declarations OR in_definition_lookup` preserves the old index membership for persisted units, while path-derived module declarations are synthesized through the existing SQL non-persisted declaration helper. Keeping this behind `OnceLock` means usage/resolution pays one batched live-OID scan only on first lookup-index use, and symbol/search paths guarded by the zero-hydration test do not touch it.
  Evidence: `tests/analyzer_sql_query_parity.rs` has the store-only lookup-index parity assertion and the existing zero-hydration guard; both pass in the core M4 suite.
- Observation: Removing resident borrowed import data required `ImportAnalysisProvider::import_info_of` to return owned `Vec<ImportInfo>` and `IAnalyzer::analyzed_files` to return owned `Vec<ProjectFile>`. This was necessary because the answers now come from live/store/transient hydration instead of references into an immutable workspace map. Callers that need stable slices keep local vectors and pass `&imports`.
  Evidence: `cargo check --all-targets --features nlp,python` is clean after updating searchtools, NLP helpers, import providers, and usage graph consumers.
- Observation: Unconditional persistent-store opening for any git-discoverable root made test runs share (and pollute) the developer repo's `.brokk/analyzer_cache.db`, because test fixtures live inside the bifrost repo. Consequence: `ruby_analyzer_test::syntax_error_file_does_not_panic` passes on the first full-suite run and fails on the second — the fixture blob is cached by run 1, hydrated by run 2, and hydrated blobs report `parse_errors = None` by contract. Root-cause fix (not a test hack): restore master's opt-in persistence split — plain constructors (`from_project`, `WorkspaceAnalyzer::build`) use the in-memory store even under git; only the explicit persisted path (`build_persisted`, service layer) opens the per-repo DB. Both contracts pinned by a new regression test (non-persisted: fresh parse twice, errors Some both times; persisted: second build hydrates with 0 parses, errors None).
  Evidence: 811KB `analyzer_cache.db` appeared at the primary repo's `.brokk/` after test runs; failing rerun output of the ruby suite; caught only because the reviewer ran the full suite a second time — single-run gates cannot see cross-run cache coupling.
- Observation: The memory benchmark must assert warm steady-state construction, not cold cache population. Cold builds still parse every file and retain parse/SQLite allocator pages, so they scale with fixture size (24.4 MB for 200 modules, 96.7 MB for 2000 modules in the current run). Running a separate warm child process after the cold child has populated the store shows the resident construction cost is near-flat (11.9 MB vs 15.0 MB), with 0 parses and 0 fresh parse-error entries at both sizes. The assertion now uses `warm_large <= warm_small * 2 + 32 MB`; the fixed slack covers SQLite mmap/page-cache and the tiny O(files) live-path map, while revived per-file resident state would push the 2000-module warm delta well past it.
  Evidence: `tests/measure_analyzer_persisted_memory.rs` prints cold and warm child-process RSS lines; explicit run reported 200 cold 24.4 MB, 200 warm 11.9 MB, 2000 cold 96.7 MB, 2000 warm 15.0 MB, warm bound 55.8 MB.
- Observation: Empty-vector flags on `code_units` only preserved the exact `HashMap` entry shape for `FileState.children` and `FileState.raw_supertypes`; post-teardown consumers treat missing entries as empty. `TreeSitterAnalyzer::raw_supertypes_of` uses `unwrap_or_default`, `IAnalyzer::direct_children` returns `unwrap_or_default`, and the remaining direct transient `FileState.children` walk in cognitive complexity simply skips absent entries the same way it skips empty vectors. Therefore the flags were removed and the store test equivalence helper now ignores empty map entries for those two maps.
  Evidence: `src/analyzer/tree_sitter_analyzer.rs` methods `raw_supertypes_of`, `direct_children`, and the function-enumeration walk around the transient `file_state.children.get(&cu)` call.
- Observation: The starred `code_units` subset CHECK `in_definition_lookup <= in_declarations` is false today. `ParsedFile::add_definition_lookup_unit` intentionally inserts lookup-only facts into `definition_lookup_units` and ranges without inserting them into `declarations`; JS/TS call this for non-public lookup support. The CHECK was skipped.
  Evidence: `src/analyzer/tree_sitter_analyzer.rs` `ParsedFile::add_definition_lookup_unit` mutates `definition_lookup_units` and `ranges`, not `declarations`; call sites include `src/analyzer/javascript/mod.rs` and `src/analyzer/typescript/mod.rs`.
- Observation: The starred persisted-kind CHECK is enumerable and stable. The on-disk `CodeUnitType` integer mapping is local to `src/analyzer/store/mod.rs`: `FileScope = 5`, `Module = 3`. The default `LanguageAdapter::should_persist_code_unit` excludes file-scope units for every language; Python, JavaScript, and TypeScript override it to exclude module units too, while other languages may persist real modules. The schema now rejects `kind = 5` for all languages and rejects `kind = 3` only for `python`, `javascript`, and `typescript`.
  Evidence: `LanguageAdapter::should_persist_code_unit` default in `src/analyzer/tree_sitter_analyzer.rs`; overrides in `src/analyzer/python/adapter.rs`, `src/analyzer/javascript/mod.rs`, and `src/analyzer/typescript/mod.rs`; code-unit kind mapping in `src/analyzer/store/mod.rs`.
- Observation: Semantic content hashes are not stored as 64-character SHA-256 hex strings in the cache schema; they are stored as 32-byte BLOB keys (`hash` and `composed_hash`). The invariant pass therefore added `CHECK(length(...) = 32)` on those BLOB key columns instead of a hex-text CHECK.
  Evidence: `src/nlp/store.rs` writes `[u8; 32]` hash arrays using `hash.as_slice()` and reads them through `decode_key_blob`.
- Observation: Quantized vector code length is runtime-variable, not a compile-time schema constant. `src/nlp/quant.rs` creates one quantizer per input vector dimension and persists `RqCode::to_bytes()`, while semantic vector tables separately store `dim`. No `length(vector) = constant` CHECK was added.
  Evidence: `quantizer(dim: usize)` is keyed by `vector.len()`, and `SemanticStore::upsert_codes` persists `dim = vector.len()` beside the encoded code bytes.

## Decision Log

- Decision: Key all parse data by git blob OID in one DB per primary repository (worktrees share it); no per-revision or "active set" layer in the database.
  Rationale: Content-addressing makes branch switches and worktrees free and avoids mirroring working-tree state into the DB, which the working tree plus `.git/index` already represent authoritatively. Follows the proven design of the semantic search cache (`src/nlp/store.rs`), minus its DB-side TEMP-table active set, which the user explicitly rejected for this store.
  Date/Author: 2026-07-02 / jbellis + Claude (planning session).
- Decision: Liveness is derived from the filesystem and `.git` on demand, never from a watcher-maintained mirror. Point queries resolve path→OID per query (index lookup + stat, hash only if stat mismatches). Workspace queries use an OID→path map rebuilt by parsing `.git/index`, memoized on the index file's own mtime/size, plus per-result stat validation of just the reported paths. The file watcher remains only an ingestion trigger for re-parsing dirty files, as it is today.
  Rationale: User is (correctly) worried about watcher drift. `.git/index` is rewritten atomically by every git operation, so keying the memo on its stat makes the map self-invalidating exactly in the scenarios watchers handle worst (checkout/rebase storms). Drift exposure for uncommitted edits is identical to the status quo.
  Date/Author: 2026-07-02 / jbellis.
- Decision: Non-git workspaces use the identical store implementation on an in-memory SQLite connection (`Connection::open_in_memory()`), not a separate HashMap-backed implementation behind a trait.
  Rationale: One real implementation instead of two; the read side is many SQL statements and mirroring them in a second impl invites divergence. Non-git loses cross-run persistence, which was already the accepted behavior (semantic search does not support non-git at all).
  Date/Author: 2026-07-02 / Claude (supersedes the pre-approval plan's trait+HashMap sketch; simplification, no behavior change).
- Decision: Rows are strictly path-independent; fully-qualified names are composed at read time from the live path via a per-language naming hook.

## Context and Orientation

bifrost is a single Rust crate (root `Cargo.toml`) providing code-analysis tools (MCP server, LSP, Python bindings). Two subsystems matter here.

The analyzer (`src/analyzer/`) parses source files with tree-sitter and answers structural queries. Key current types, all in `src/analyzer/tree_sitter_analyzer.rs` unless noted: `FileState` (~line 150) holds one file's extracted results — the full `source: String`, declarations as `CodeUnit`s, per-unit `ranges`, `signatures`, `signature_metadata`, `children`, `raw_supertypes`, `imports`, `type_identifiers`, `contains_tests`. The raw tree-sitter `Tree` is already discarded after extraction. `AnalyzerState` (~line 175) is the per-language whole-workspace container: `files: HashMap<ProjectFile, FileState>` plus derived cross-file indexes (`definitions` mapping FQ name → units, `definition_lookup_index`, `children`, `module_children`, `classes_by_package`, …), held as `Arc<AnalyzerState>` inside `TreeSitterAnalyzer<A>`. A `CodeUnit` (`src/analyzer/model.rs` ~line 364) is one declaration: file, kind, package name, short name, optional signature, behind an `Arc`. A `ProjectFile` (`model.rs` ~line 245) is a normalized (root, relative path) pair. All queries go through the `IAnalyzer` trait (`src/analyzer/i_analyzer.rs`); consumers are `src/searchtools.rs` (MCP tools) and `src/lsp/handlers/*`. Usage/graph analysis (`src/analyzer/usages/`) re-parses files on demand and drops trees — it never reads `FileState`s and is out of scope. The existing disk cache (`src/analyzer/persistence/`) writes one `analyzer.db` per worktree under `.bifrost/`, keyed by relative path + (mtime, size), storing bincode payloads (`persistence/payload.rs`, `PAYLOAD_VERSION 3`) that are bulk-hydrated into `AnalyzerState` at startup; `reconcile::plan` decides parse-vs-hydrate; `epoch.rs` computes an `analysis_epoch` from grammar/query/crate versions that invalidates everything when bumped. This whole per-worktree layer is retired by Milestone 4. There is also `search_definitions_persisted` (`i_analyzer.rs`, impl in `tree_sitter_analyzer.rs` ~line 1614) which already queries that DB directly for symbol search — precedent for SQL-backed reads, replaced in Milestone 3.

The semantic search index (`src/nlp/`) is the design template. `src/nlp/store.rs` opens `.brokk/semantic_cache.db` at the primary repo root — `semantic_db_path` resolves a linked worktree back to the primary repo via `repo.commondir().parent()` — with WAL, `synchronous=NORMAL`, `foreign_keys=ON`, incremental autovacuum, and treats the DB as a rebuildable cache: on any `schema_version` mismatch it drops and recreates all tables (no migrations). `src/nlp/gitcache.rs` derives working-tree blob OIDs: clean files take the OID straight from the git index entry (no file read); dirty/untracked files are hashed from disk with `Oid::hash_file`; `working_tree_oids_targeted` resolves just a named set of paths in O(changed); `reachable_oids`-style helpers stream `git rev-list --objects --all` into a `GrowableBloom` bloom filter for GC (union'd with every worktree's uncommitted OIDs; bloom filters have no false negatives so reachable blobs are never dropped). `src/nlp/indexer.rs` shows the "materialize only missing blobs" pattern (`materialize_missing`, ~line 500): resolve working tree to OIDs, ask the store which are unseen, process only those. `rusqlite` 0.32 with the `bundled` feature and `git2` 0.20 are already dependencies (root `Cargo.toml`).

Term definitions. "Blob OID": git's SHA-1 of a file's content; identical content ⇒ identical OID regardless of path, branch, or worktree. "Primary repository root": the main checkout's directory that owns `.git`; linked worktrees created by `git worktree add` share its object database. "Live" blob: a blob that is the current content of some working-tree file in the worktree being queried. "Path-independent": a stored row may not contain any data derived from the file's path (names derived from directories, `__init__.py` probing, etc.).

    pub struct Liveness { /* repo handle + memoized snapshot */ }

        /// Point resolution: index entry OID if stat matches, else hash from disk.
        pub fn oid_for_path(&self, file: &ProjectFile) -> anyhow::Result<Option<git2::Oid>>;

    }

## Idempotence and Recovery

The DB is a rebuildable cache: deleting `.brokk/analyzer_cache.db` is always safe and merely costs a re-parse. Schema/epoch mismatches wipe and rebuild by design. All writes are `INSERT OR IGNORE` of content-addressed rows, so re-running any build/update step is harmless. Each milestone lands as one or more compiling, test-green commits on this branch; if a milestone goes wrong, `git reset` to the previous checkpoint loses nothing but that milestone.
